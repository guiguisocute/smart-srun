# SRun 认证页诊断采集说明

适用于排查新增学校时，浏览器手动登录可用、但 smart-srun 登录不可用的情况。

当前诊断脚本：`scripts/srun_portal_diagnostic.user.js`

脚本版本 `0.1.2` 起会把已捕获事件临时保存在同一认证网关 origin 的 `localStorage` 中，两小时内回到该认证页可以恢复日志。这个设计是为了处理“登录成功后立刻跳转学校官网，来不及点复制”的情况。

## 需要同学准备

- 一台能打开校园网认证页的电脑
- Chrome / Edge / Firefox
- Tampermonkey（油猴）浏览器扩展
- 校园网账号密码

## 安装脚本

1. 安装 Tampermonkey（油猴）扩展。
2. 打开 Tampermonkey 管理面板，点击“添加新脚本”。
3. 删除默认模板内容。
4. 复制 `scripts/srun_portal_diagnostic.user.js` 的全部内容，粘贴进去。
5. 保存脚本，并确认脚本处于启用状态。

## 采集日志

1. 连接到需要认证的黄山学院校园网、有线网或无线网。
2. 打开认证页：

   ```text
   http://172.16.2.14/srun_portal_pc?ac_id=1
   ```

3. 页面右下角应出现“Smart SRun 诊断采集”小面板。
4. 像平时一样在认证页手动输入账号密码并登录。
5. 登录成功或失败后，点击右下角小面板里的“复制诊断日志”。
6. 把复制出来的整段 JSON 日志发给维护者。

也可以点击“下载 JSON”，把下载到的 `.json` 文件发给维护者。

如果登录成功后立刻跳转到了学校官网，看不到右下角面板：

1. 不要清浏览器缓存，也不要关闭该浏览器窗口。
2. 在地址栏重新打开原来的认证网关页面，例如 `http://172.16.2.14/srun_portal_pc?ac_id=1`。
3. 右下角面板如果显示“已恢复上次捕获”，直接点“复制诊断日志”或“下载 JSON”。
4. 如果学校网关自动把你再次跳走，可以先断开校园网再重连，或打开无痕窗口重新进入认证页后再采集一次。

## 已知配置坑：认证地址只填裸网关

插件里的“认证地址”字段目前应填写认证网关的裸地址，不要填写浏览器地址栏里的完整页面路径。

以黄山学院为例，插件中应填写：

```text
http://172.16.2.14
```

不要填写：

```text
http://172.16.2.14/srun_portal_pc
http://172.16.2.14/srun_portal_pc?ac_id=1
```

原因是插件会基于认证地址自动拼接接口路径，例如 `/cgi-bin/get_challenge`、`/cgi-bin/srun_portal` 和 `/cgi-bin/rad_user_info`。如果把完整页面地址填进去，实际请求路径会被拼错，常见表现是 challenge 阶段反复报：

```text
响应解析错误: Expecting value: line 1 column 1 (char 0)
```

当前版本已在 LuCI 保存和运行时解析两侧做了规范化：保存校园网账号配置时会把常见 Portal 页面 URL 规范化成裸网关，例如把 `http://172.16.2.14/srun_portal_pc?ac_id=1` 规范化为 `http://172.16.2.14`。

## #18/#19 当前判断

- #18 黄山学院：现有诊断日志只显示脚本加载在 `http://172.16.2.14/srun_portal_pc_success`，未捕获 challenge/login 请求，因此还不能判定是算法问题。更可能是已经处于登录成功页或登录后跳转过快，需要按上面的持久化恢复流程重新抓一次从认证页开始的完整请求。
- #19 临沂大学：HAR 里的 `10.232.x.x` 更像用户当时的本机源 IP 或网关分配地址，不像目标认证网关。`wget --bind-address=10.232.49.136` 里的 bind-address 表示本机发包源地址，不是插件要访问的目标网关；目标网关仍应是裸 origin，例如 `http://10.88.1.20`。
- `xn` 免后缀不是 #18/#19 的主要矛盾。默认学校元数据已经把 `xn` 作为免后缀运营商处理；真正需要核查的是认证网关 origin、`ac_id`、运营商后缀，以及浏览器实际提交的登录字段。

## 运营商后缀能否探测

可以“推断”或“验证”，但不能只靠一次无密码探测准确判断。

- 最可靠来源是浏览器实际登录请求里的 `username`。诊断脚本会打码账号主体，但保留 `@后缀`，所以用户通过网页登录成功一次后，维护者通常能看到实际提交的是 `学号@cmcc`、`学号@hcmcc`，还是纯学号。
- `/cgi-bin/get_challenge` 主要按账号和 IP 发 challenge，通常不能证明某个后缀一定可登录。
- 真正验证后缀是否可用，需要带账号密码请求 `/cgi-bin/srun_portal`。这相当于一次登录尝试，可能影响当前在线状态，也可能触发学校侧风控，因此不建议后台静默批量尝试。
- 如果以后要做插件内探测，建议做成用户主动点击的“后缀验证”动作：只测试学校预设给出的少量候选后缀，过程中明确提示会发起登录尝试，成功后立即停止，并且日志继续打码账号密码。

## 隐私说明

诊断脚本默认不会导出明文密码，也不会导出 challenge token 和 `{SRBX1}` 的 `info` 原文。

导出的日志会包含：

- 认证网关地址、接口路径、`ac_id`、`ip`、`n`、`type`
- 登录请求中顶层 `username` / `password` 是否为空
- 浏览器本地解码后的 `info` 结构摘要，其中账号会打码、密码只显示是否存在和长度
- 登录接口返回的错误码和错误信息
- `chksum` 是否符合浏览器实际使用的顶层字段签名规则

如果后续需要进一步排查，维护者可能会再请求更详细的临时日志；在那之前，不建议公开发送完整登录 URL。

## 可以转发给 issue 作者的话

请先安装油猴脚本，然后在电脑浏览器里打开：

```text
http://172.16.2.14/srun_portal_pc?ac_id=1
```

页面右下角出现“Smart SRun 诊断采集”后，手动登录一次。登录结束后点“复制诊断日志”，把复制出来的 JSON 发给我即可。

这个日志默认会打码账号、密码、challenge token 和加密的 `info` 原文，主要用于确认黄山学院的深澜页面具体使用了哪种登录参数格式。

如果需要在插件里复测，请注意“认证地址”只填：

```text
http://172.16.2.14
```

不要把 `/srun_portal_pc?ac_id=1` 一起填进去。
